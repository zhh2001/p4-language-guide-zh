# 08 · 匹配-动作表（Match-Action Table）

> 本章目标：透彻理解 P4 最核心的抽象——**表**。P4 的威力几乎都凝结在这一个概念里。

## 8.1 为什么要有"表"

数据平面的本质工作：**"看包头里的这些字段，匹配规则，执行对应动作。"**

如果写死：

```p4
if      (ip.dst == 10.0.0.1) out = 1;
else if (ip.dst == 10.0.0.2) out = 2;
else if (ip.dst == 10.0.0.3) out = 3;
else ... // 好几万条
```

显然不可行——规则是动态的、海量的。于是 P4 把它抽象成：

> 一张 **匹配-动作表**：源码里只定义 **"要匹配哪些字段、可能执行哪些动作"**；具体表项由 **控制平面** 实时下发。

## 8.2 匹配-动作单元的数据流

```text
                            ┌────────────────┐
  headers / metadata ─────► │  构造 Key      │
                            └─────┬──────────┘
                                  ▼
                        ┌──────────────────┐
                        │ 在查找表中匹配    │
                        │ ┌──────────────┐ │     ◀── 控制平面
                        │ │  Entry 1     │ │         下发/删除
                        │ │  Entry 2     │ │
                        │ │  ...         │ │
                        │ └──────────────┘ │
                        └──────┬───────────┘
                               │ (hit ⇒ action + params)
                               │ (miss ⇒ default_action)
                               ▼
                        ┌──────────────────┐
                        │ 执行选中的动作    │ ─► 写入 headers / metadata
                        └──────────────────┘
```

## 8.3 表的最小结构

```p4
table ipv4_lpm {
    key = {
        hdr.ipv4.dstAddr : lpm;
    }
    actions = {
        set_nhop;
        drop;
        NoAction;
    }
    size           = 1024;
    default_action = drop;
}
```

四要素：

1. **`key`** —— 查找时构造的匹配键
2. **`actions`** —— 表允许的动作清单
3. **`default_action`** —— 未命中时执行（可选；未写 = `NoAction`）
4. **`size`** —— 期望容量（可选）

## 8.4 `key`：匹配什么、怎么匹配

语法：`表达式 : match_kind;`

### 8.4.1 预定义的三种 match_kind

`core.p4` 里定义：

```p4
match_kind { exact, ternary, lpm }
```

| match_kind | 含义 | 典型场景 |
| ---------- | ---- | -------- |
| `exact`    | 完全相等 | dstMAC → 端口、VLAN ID |
| `lpm`      | 最长前缀匹配 | IPv4/IPv6 路由 |
| `ternary`  | 值 + 掩码（任意位"不关心"） | ACL |

### 8.4.2 架构特定的 match_kind

V1Model 还加了 `selector` 和 `range`：

```p4
key = {
    hdr.ipv4.dst : lpm;
    hdr.tcp.port : range;      // v1model 扩展
    meta.group   : selector;   // 与 action_selector 配合
}
```

### 8.4.3 多字段组合

一张表可以有多个 key：

```p4
table acl {
    key = {
        std_meta.ingress_port : ternary;
        hdr.ethernet.srcAddr  : ternary;
        hdr.ipv4.srcAddr      : lpm;
        hdr.ipv4.protocol     : exact;
    }
    actions = { permit; deny; NoAction; }
    size = 512;
    default_action = permit;
}
```

这类多字段 + 混合匹配的表，通常需要**表项优先级**。

### 8.4.4 空 key 表

```p4
table always_run {
    key = { }
    actions = { do_x; }
    default_action = do_x;
}
```

没有查找表，只执行 default_action。相当于"每次都运行这个动作"——有时用来集中管理某个独立步骤。

## 8.5 `actions`：合法动作清单

```p4
action drop()                 { mark_to_drop(std_meta); }
action set_nhop(bit<32> nh,
                bit<9>  port) {
    meta.nextHop            = nh;
    std_meta.egress_spec    = port;
    hdr.ipv4.ttl            = hdr.ipv4.ttl - 1;
}

table ipv4_lpm {
    key     = { hdr.ipv4.dstAddr: lpm; }
    actions = { set_nhop; drop; NoAction; }
    /* ... */
}
```

### 8.5.1 参数绑定规则

- **有方向**（`in`/`out`/`inout`）参数：**必须在动作列表里绑定为具体值**
- **无方向** 参数：**不能**在动作列表里绑定，由控制平面下发

错例：

```p4
action a(in bit<32> x) { /* ... */ }
bit<32> z;
action b(inout bit<32> x, bit<8> data) { /* ... */ }

table t {
    actions = {
        a;         // ❌ in 参数 x 没绑定
        a(5);      // ✅ 绑定 x = 5
        b(z);      // ✅ inout x 绑定到全局变量 z
        b(z, 3);   // ❌ 无方向参数 data 不能在这里绑定
        b();       // ❌ x 没绑定
    }
}
```

### 8.5.2 `@tableonly` 和 `@defaultonly`

限制 action 能出现在 `entries` 里还是 `default_action` 里：

```p4
action a() { /* ... */ }
@tableonly    action b() { /* 只能在表项里 */ }
@defaultonly  action c() { /* 只能作默认动作 */ }

table t {
    actions = { a; b; c; }
    default_action = c;    // ✅
    // default_action = b;  ❌
}
```

## 8.6 `default_action`：未命中时的兜底

```p4
default_action = drop;              // 合法
const default_action = drop;        // 控制平面不能改
default_action = set_nhop(0, 511);  // 带参数
```

- 必须是 `actions` 列表里出现过的动作
- 传入参数必须和 `actions` 列表里"绑定形式"一致

未显式设置时，编译器会默认加 `NoAction`。

## 8.7 编译期表项 `entries`

表项通常由控制平面在运行时下发，但也可以在**编译时**固化一组：

```p4
table t_exact_ternary {
    key = {
        h.h.e : exact;
        h.h.t : ternary;
    }
    actions = { a; a_params; }
    default_action = a;

    const entries = {
        (0x01, 0x1111 &&& 0xF     ) : a_params(1);
        (0x02, 0x1181             ) : a_params(2);
        (0x03, 0x1111 &&& 0xF000  ) : a_params(3);
        (0x06, _                  ) : a_params(6);
        _                           : a;
    }
}
```

### `const entries` 与 `entries` 的区别

| 形式 | 控制平面能否增删改 |
| ---- | ------------------ |
| `const entries = { ... }` | **不能**（整张表锁死） |
| `entries = { ... }` | 默认**可**；每条表项前加 `const` 单独锁 |

固化表项的好处：

- 编译器知道表的"实际规模"，可能给出更紧凑的硬件资源分配
- 实现固定算法（比如 parser 里用来分发的表）更清晰

## 8.8 表项优先级

**全部字段都是 `exact` 或 `lpm`**：不用指定优先级——要么键唯一（exact）、要么前缀长度决定（lpm）。

**只要有 `ternary` 字段**：就必须有优先级——同一个输入可能同时"匹配多条"，优先级决定谁赢。

### 8.8.1 两种取向

| 表属性 | 含义 | 规则 |
| ---- | ---- | ---- |
| `largest_priority_wins = true`（默认） | "值大者胜" | 优先级为正整数，越大越先匹配 |
| `largest_priority_wins = false` | "值小者胜" | 优先级 ≥ 0，越小越先匹配 |

### 8.8.2 `priority_delta`

配合 `entries` 使用——未显式指定优先级的连续表项，自动按这个步长递增。默认 `1`。

```p4
table t_acl {
    key = { ... }
    actions = { ... }
    largest_priority_wins = false;
    priority_delta = 10;
    @noWarn("duplicate_priorities")
    entries = {
        const priority=10: (...) : a_params(1);
                           (...) : a_params(2);   // priority=20
                           (...) : a_params(3);   // priority=30
        const              (...) : a_params(4);   // priority=40
              priority=40: (...) : a_params(5);
                           (...) : a_params(6);   // priority=50
    }
}
```

留"间隙"是为了后续控制平面能**插队**。

## 8.9 表的调用与返回值

`table.apply()` 返回一个 struct：

```text
struct apply_result_T {
    bool hit;
    bool miss;           // == !hit
    enum_T action_run;   // 枚举：实际执行的 action
}
```

**用法示例**

```p4
// ① 只关心是否命中
if (dmac.apply().hit) {
    forward_known();
} else {
    flood();
}

// ② 与 LPM 回退
if (host_exact.apply().miss) {
    ipv4_lpm.apply();
}

// ③ 根据实际 action 分支
switch (acl.apply().action_run) {
    deny:   { return; }
    permit: { /* fall through */ }
    default: { /* ... */ }
}
```

## 8.10 表调用的伪代码

理解一下底层到底发生了什么：

```text
apply_result t.apply():
    lookupKey = buildKey(t.key)          # 按声明顺序串起来
    RA        = t.lookup(lookupKey)      # 命中 / 未命中
    if RA == null:
        result.hit = false
        RA = t.default_action
    else:
        result.hit = true
    result.miss       = !result.hit
    result.action_run = typeof(RA)
    bind_args(RA)                        # 绑定控制平面参数
    execute(RA)                          # 真正运行动作
    return result
```

## 8.11 一张表的完整例子

```p4
action drop()                    { mark_to_drop(std_meta); }
action set_egress(bit<9> port)   { std_meta.egress_spec = port; }
action set_nhop(bit<32> nh_ip,
                bit<9>  port) {
    meta.nhop_ip          = nh_ip;
    std_meta.egress_spec  = port;
    hdr.ipv4.ttl          = hdr.ipv4.ttl - 1;
}

table ipv4_lpm {
    key = {
        hdr.ipv4.dstAddr : lpm;
    }
    actions = {
        set_nhop;
        drop;
        NoAction;
    }
    size           = 2048;
    default_action = drop;
}
```

## 8.12 表的约束清单

| 约束 | 原因 |
| ---- | ---- |
| `key` 是编译期常量（表达式列表） | 硬件需预知要匹配哪些字段 |
| `size` 是期望值，实际容量看目标 | 目标决定硬件资源分配 |
| 表项由控制平面下发（除非 `const entries`） | 数据平面只关心"形状" |
| 一个 action 只能绑一次 | 避免歧义 |
| 不能在 action 的参数里调用 `table.apply` | 保证表的语义可静态分析 |

## 8.13 本章小结

- **表 = 只声明"形状"，规则由控制平面下发**
- 三种 match_kind：`exact`、`lpm`、`ternary`；架构可扩展（v1model 加了 `range`、`selector`）
- 有方向参数要在 `actions` 里绑定；无方向参数留给控制平面
- `const entries` 能编译时固化表项，甚至锁死表
- ternary 表必须定优先级；默认"大者胜"
- `table.apply()` 返回 `hit`/`miss`/`action_run`，能驱动后续控制流
