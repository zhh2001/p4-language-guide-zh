# 02 · P4 概述与核心概念

> 本章目标：理解 **P4 是什么、它解决了什么问题**，以及它在整张 SDN 版图中的位置。不写一行代码，但这是你接下来所有学习的"心法"。

## 2.1 P4 是什么

P4 的全称是 **Programming Protocol-independent Packet Processors**——"编程协议无关的报文处理器"。它是一门 **领域特定语言（DSL）**，专门用来描述 **网络设备的数据平面**（转发平面）该如何处理每一个报文。

一句话总结：

> **P4 告诉交换机"该如何解析、匹配、修改、转发每一个包"。**

它 **不是** 替代 C 或 Python 的通用编程语言，**也不是** 替代 Open vSwitch 配置的管理工具。它只关心一件事：报文在硬件里到底怎么走。

## 2.2 没有 P4 之前，世界长什么样

传统网络设备有两个严重问题：

### 问题 1：固定功能流水线（FFP）

交换机 ASIC 的流水线在流片时就被写死。想支持一个新的协议（例如 VXLAN、MPLS、INT），要么：

- 等厂商出下一代芯片 → 18 个月起步
- 写 **微码（microcode）** 打补丁 → 需厂商授权 + 性能损失

于是学术界的新想法永远被"没有硬件支持"卡死。

### 问题 2：南向接口暴露的能力有限

OpenFlow 1.0 定义了 12 个匹配字段，1.3 扩展到 40 多个。每次新增协议就要扩协议号——像在给有限的抽屉手动编号，本质上还是**硬件决定你能编程什么**。

P4 要做的事情，是把这个关系 **倒过来**：**你写代码决定硬件的流水线长什么样**。

## 2.3 P4 的三大设计目标

### 目标 1：协议无关（Protocol-Independent）

编译器不预设任何协议。想匹配以太网就自己写 `header Ethernet_h`，想匹配 VXLAN 就自己写 `header VXLAN_h`。**P4 程序本身定义了这台交换机理解哪些协议。**

### 目标 2：目标无关（Target-Independent）

同一份 P4 源码，配不同的 "architecture" 后端就能编到：

- BMv2（软件仿真，x86）
- Tofino（ASIC，线速 Tbps）
- DPDK（用户态快速转发）
- eBPF（Linux 内核）
- FPGA / 智能网卡

### 目标 3：可重构（Reconfigurable）

想加功能？改 P4 源码，重新编译、重新加载流水线。生产网络里，这可以在毫秒级完成——无需换设备。

## 2.4 PISA：P4 背后的硬件抽象

P4 所描述的流水线，对应一个叫 **PISA（Protocol-Independent Switch Architecture）** 的抽象硬件模型：

```text
           ┌─────────────────────────────────────────────────────┐
  Packet  ─►│  Programmable   │  Programmable   │  Programmable   │─► Packet
           │     Parser       │  Match-Action   │    Deparser     │
           │                  │     Pipeline    │                 │
           └─────────────────────────────────────────────────────┘
                   ▲                   ▲                  ▲
                   │                   │                  │
             你写的代码         你写的代码         你写的代码
           (解析哪些报头)     (查哪些表、改哪些字段)    (重新拼装报文)
```

三个可编程组件，外加一些固定功能（CRC、队列、调度），构成了真实芯片的"骨架"。P4 程序其实就是在填这三块"空白"。

## 2.5 一份 P4 程序由什么组成

一个典型的 P4_16 程序大致长这样：

```p4
#include <core.p4>        // P4 核心库（extract、verify 等原语）
#include <v1model.p4>     // 架构描述（不同架构不同）

// 1) 定义数据类型 —— 要解析和处理哪些报头
header Ethernet_h { ... }
header IPv4_h     { ... }
struct headers    { Ethernet_h ethernet; IPv4_h ipv4; }

// 2) Parser —— 告诉交换机如何从原始字节里抽出报头
parser MyParser(packet_in p, out headers hdr, ...) { ... }

// 3) Match-Action 流水线 —— 核心转发逻辑
control MyIngress(inout headers hdr, ...) {
    action forward(bit<9> port) { ... }
    table ipv4_lpm { key = { ... } actions = { ... } }
    apply { ipv4_lpm.apply(); }
}

// 4) Deparser —— 把处理过的报头重新拼回字节流
control MyDeparser(packet_out p, in headers hdr) { ... }

// 5) 实例化顶层包，让上面所有块"装"进架构里
V1Switch(MyParser(), ..., MyIngress(), ..., MyDeparser()) main;
```

看不懂没关系，这只是让你建立"程序形状"的感知。每一部分我们都会在后续章节逐个深入。

## 2.6 P4_14 vs P4_16：你该学哪个？

- **P4_14**（2014）：第一代，语法更接近配置脚本，已被社区放弃演进
- **P4_16**（2016–至今）：第二代，语法更现代、语义更严谨、目标无关性更好

**只学 P4_16**。本教程所有内容都基于 P4_16。

遗留系统中还有 P4_14 代码？P4 官方提供 `p4test` 带自动转换，一般用不到你手动写。

## 2.7 P4 在 SDN 栈中的位置

```text
┌────────────────────────────────┐
│       应用层（App）            │  比如：流量工程、DDoS 检测
├────────────────────────────────┤
│    控制平面（Controller）      │  ONOS、SONiC、自研 Python 脚本
│     ↕ 南向接口：gRPC           │
├────────────────────────────────┤
│    P4Runtime / 配置接口        │  API 标准：P4Runtime、gNMI、gNOI
├────────────────────────────────┤
│  数据平面 (P4 程序描述的流水线)  │  ← 你要学的就是这一层
├────────────────────────────────┤
│         硬件 / 软件目标        │  Tofino / BMv2 / DPDK / FPGA
└────────────────────────────────┘
```

- **P4 语言** 描述数据平面的 **转发逻辑**（流水线形状 + 匹配/动作规则）
- **P4Runtime** 是控制平面向数据平面 **下发表项** 的标准 API（gRPC）
- **p4info** 是编译器产出的"元数据清单"，告诉控制平面这份 P4 程序里有哪些表、哪些动作

这三个概念我们会在后面的章节里反复碰到，现在先混个脸熟。

## 2.8 几个高频混淆点

| 常见误解 | 实际情况 |
| -------- | -------- |
| "P4 是 OpenFlow 的下一代" | P4 **包含但不限于** OpenFlow 的能力；OpenFlow 只是一个控制协议，P4 还改变了数据平面本身 |
| "P4 用于写控制平面" | 错。P4 **只** 写数据平面；控制平面用通用语言（Python/Go）+ P4Runtime |
| "P4 是硬件描述语言" | P4 不是 Verilog / VHDL；它描述 **转发行为**，不描述门电路 |
| "写 P4 就能跑在 Tofino 上" | 仅编译通过只是第一步，要跑在真机上还需要 SDE、驱动、许可等 |

## 2.9 学 P4 的"心智切换"

从传统编程切换到 P4，你需要习惯几件"反常识"的事：

1. **没有循环、没有递归**——流水线必须静态展开（硬件必须知道最大深度）
2. **没有浮点、没有指针**——一切都是位串
3. **没有全局可变状态**——要存状态，必须用 `register` 这样的 extern
4. **每个报文"流"过一遍流水线**——不是"for each packet do X"，而是"pipeline always does X"
5. **表是控制平面填的**——源码里只有表的"形状"，具体匹配规则由 Runtime 动态下发

如果你带着写 Python 的直觉来，会很不适应。请把 P4 当成"**描述一块电路的行为**"来写。

## 2.10 本章小结

- P4 是一门 **描述数据平面转发行为** 的领域特定语言
- 它围绕 **PISA 模型**：Parser → Match-Action → Deparser
- 三大目标：**协议无关、目标无关、可重构**
- 只学 **P4_16**；控制平面用 **P4Runtime**
- 心智要切换到 **"流水线"** 而不是 **"程序"** 的视角
