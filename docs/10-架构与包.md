# 10 · 架构与包（Architecture & Package）

> 本章目标：理解 P4_16 里最**容易被跳过却最关键**的一块——架构模型。不懂架构，你的 P4 程序就是一堆没法组装到硬件上的零件。

## 10.1 为什么要有"架构"

P4_16 做了一个大胆的决定：**语言本身不假定任何一台具体的交换机模型**。

- 交换机/智能网卡厂商差异极大
- 一些芯片有 ingress/egress 两阶段，另一些是单阶段流水线
- 有的芯片内建队列、镜像、重循环等操作，有的则由 CPU 承担

如果 P4 语言把"pipeline 里就是 ingress + egress"这种模型硬编码进去，就绑死了。

于是 P4 规范只保留**通用语法**，把"具体这台交换机长什么样"抽离成一份 **架构描述文件**（Architecture Description）——通常是 `v1model.p4`、`psa.p4`、`tna.p4` 等。

## 10.2 架构描述做了什么

一份架构描述文件里包含：

1. **预定义的常量、错误、元数据类型**（例如 `standard_metadata_t`）
2. **可编程块的接口（类型声明）**
   - `parser Parser<H>(packet_in b, out H h, ...);`
   - `control Ingress<H>(inout H h, ...);`
   - `control Deparser<H>(packet_out b, in H h);`
3. **顶层包（Package）声明**
   - `package V1Switch<H, M>(Parser<H, M> p, ..., Deparser<H> d);`
4. **架构提供的 extern 对象**
   - 计数器、寄存器、哈希、校验和等

**关键点**：架构描述只声明"**有哪些可编程位置、它们的接口是什么**"，**不写**具体实现。实现由用户的 P4 程序填入。

## 10.3 你的 P4 程序和架构的关系

```text
        ┌────────────────────────────┐
        │   架构描述文件              │
        │   (e.g. v1model.p4)         │
        │                             │
        │   parser  Parser<H>(...);   │    ← 声明接口
        │   control Ingress<H>(...);  │
        │   control Deparser<H>(...); │
        │   package V1Switch<H,M>     │
        │     (Parser<H> p,           │
        │      Ingress<H> ig, ...);   │
        └────────────────────────────┘
                      ▲
                      │  用户实现并装配
                      │
        ┌────────────────────────────┐
        │   你的 P4 程序              │
        │                             │
        │   parser  MyParser(...) {...}│
        │   control MyIngress(...) {...}│
        │   control MyDeparser(...) {...}│
        │                             │
        │   V1Switch(MyParser(),      │
        │            MyIngress(),     │
        │            MyDeparser()) main;│
        └────────────────────────────┘
```

两件事：

1. 你要**实现**架构声明的每个 control/parser 接口
2. 你要**实例化**顶层包，命名为 `main`——这是整个 P4 程序的入口

## 10.4 最小架构示例

规范里给了一个简化架构示例，便于理解"架构是一份什么样的文件"：

```p4
// 1. 声明可编程块的类型
parser  Parser<IH>(packet_in b, out IH parsedHeaders);

// ingress 匹配-动作流水线
control IPipe<T, IH, OH>(in  IH inputHeaders,
                         in  InControl inCtrl,
                         out OH outputHeaders,
                         out T  toEgress,
                         out OutControl outCtrl);

// egress 匹配-动作流水线
control EPipe<T, IH, OH>(in  IH inputHeaders,
                         in  InControl inCtrl,
                         in  T  fromIngress,
                         out OH outputHeaders,
                         out OutControl outCtrl);

control Deparser<OH>(in OH outputHeaders, packet_out b);

// 2. 声明子包 Ingress 和 Egress
package Ingress<T, IH, OH>(Parser<IH> p,
                           IPipe<T, IH, OH> map,
                           Deparser<OH> d);
package Egress<T, IH, OH>(Parser<IH> p,
                          EPipe<T, IH, OH> map,
                          Deparser<OH> d);

// 3. 声明顶层包
package Switch<T>(Ingress<T, _, _> ingress, Egress<T, _, _> egress);
```

用户程序：

```p4
parser MyParser(packet_in b, out user_headers_t h) { /* ... */ }
control MyIngressPipe(/* ... */) { /* ... */ }
control MyEgressPipe(/* ... */) { /* ... */ }
control MyDeparser(in user_headers_t h, packet_out b) { /* ... */ }

// 把 Parser + Pipe + Deparser 装进子包
Ingress(MyParser(), MyIngressPipe(), MyDeparser()) ingressPipe;
Egress (MyParser(), MyEgressPipe(),  MyDeparser()) egressPipe;

// 实例化顶层包
Switch(ingressPipe, egressPipe) main;
```

## 10.5 `package` —— 架构的"装配底座"

`package` 是 P4_16 的独有概念：**一个无实现的容器类型**，只规定"我要收哪些类型的子块"。

```p4
package V1Switch<H, M>(Parser<H, M> p,
                      VerifyChecksum<H, M> vr,
                      Ingress<H, M>  ig,
                      Egress<H, M>   eg,
                      ComputeChecksum<H, M> ck,
                      Deparser<H>    dep);
```

这就像 C++ 的模板类 + 构造函数——`V1Switch<headers, metadata>(parser_instance, ..., deparser_instance) main;` 填入具体实例就完成了装配。

### 10.5.1 `package` 的约束

- **构造参数必须无方向**（这些参数是在编译 / 装配时固化的）
- **不能有 body**——没有 `apply` 块，没有 action，没有表
- 可以嵌套（用一个 package 做另一个 package 的参数）

## 10.6 `main`：P4 程序的唯一入口

```p4
V1Switch(MyParser(), MyVerify(), MyIngress(), MyEgress(), MyCompute(), MyDeparser()) main;
```

- `main` 是顶层命名空间里的特殊标识符——编译器就认它
- 类型必须是架构描述里声明的某个 `package`
- 所有参数必须一次性提供（除非用 `@optional`）

### 10.6.1 没有 `main` 会怎样？

编译器报错：

```text
error: Program does not contain a top-level module 'main'
```

### 10.6.2 两个包冲突？

一个 P4 程序只能对应**一个** `main`，也就对应**一个**架构。想同时支持多架构，要写多份 P4 源文件或用 `#ifdef`。

## 10.7 架构决定了什么

架构本身会给你以下约束：

| 约束来源 | 例子 |
| -------- | ---- |
| 提供哪些 extern | V1Model 有 `register`；有的 PSA 实现不允许 |
| 元数据字段 | V1Model 用 `standard_metadata_t` |
| 支持哪些 match_kind | V1Model 扩展了 `range` |
| Pipeline 形状 | V1Model 是 ingress+egress；VSS 是单级 |
| 校验和接口 | 独立控制块 vs 手动 Checksum16 |

**这也是为什么一份 P4 程序不能"写一次跑所有目标"**——语言可移植，但架构不一定。

## 10.8 常见架构一览

| 架构 | 描述文件 | 对应目标 | 状态 |
| ---- | -------- | -------- | ---- |
| VSS | `very_simple_switch_model.p4` | 教学 / 规范附录 | 规范参考 |
| V1Model | `v1model.p4` | BMv2 `simple_switch` | 最常用（学习/教学） |
| PSA | `psa.p4` | 任何遵从 PSA 的实现 | 标准化中（生产推荐） |
| TNA | `tofino.p4`（闭源） | Intel Tofino | 生产（商业） |
| PNA | `pna.p4` | 智能网卡（NIC） | 新兴 |

本教程重点讲 **V1Model**（下一章）。

## 10.9 参数化与构造参数

`parser` / `control` 除了"运行时参数"，还可以有 **构造参数**：

```p4
parser GenericParser<H>(packet_in b, out H hdr)
                       (bool udpSupport) {   // 构造参数列表：()
    state tryudp {
        transition select(udpSupport) {
            false: accept;
            true:  udp;
        }
    }
    /* ... */
}
```

实例化时必须提供构造参数：

```p4
GenericParser<my_headers_t>(false) ingressParser;
```

构造参数的特征：

- **必须无方向**
- **只能在实例化时赋值**，一旦装配就固化

## 10.10 直接调用（语法糖）

对于**没有构造参数**的 control / parser，可以不显式实例化：

```p4
control Callee(/* ... */) { /* ... */ }

control Caller(/* ... */) {
    apply {
        Callee.apply(/* ... */);   // 等价于：Callee() c; c.apply(...);
    }
}
```

编译器自动帮你生成局部实例。

带泛型参数也可以：

```p4
control C<T>(/* ... */) { /* ... */ }

control Top(/* ... */) {
    apply {
        C<bit<32>>.apply(/* ... */);
    }
}
```

一个 control 多次直接调用：

- **不同作用域**：生成不同的局部实例
- **同一作用域**：生成不同实例，但 `@name` 共享同一全局名

## 10.11 本章小结

- P4_16 **故意把硬件模型解耦** 到架构描述文件
- 架构描述 = 接口 + extern + 顶层 package 声明
- 你的任务：实现每个 block + 实例化 `main`
- `package` 是一个编译期"装配容器"，只接收无方向参数
- 每个 P4 程序对应一个架构、一个 `main`
