# 07 · 控制块与动作

> 本章目标：掌握 P4 的 `control` 块和 `action` 声明——这是数据平面真正"做决策"的地方。下一章 [08 · 匹配-动作表](./08-匹配动作表.md) 会接着讲如何用表把动作串起来。

## 7.1 控制块是什么

Parser 把字节流变成了 headers。现在，control 块要：

- **读写** 这些 headers 和 metadata
- **查表**，根据表项决定执行哪个 action
- **调用其他 control 块**（构成可复用的子模块）

简单说：**control 就是你真正写转发逻辑的地方**。

## 7.2 control 的约束

相比 parser 和 function，control 有几条重要约束：

1. **不能实例化 parser**
2. **没有异常控制流**——parser 的 `verify`、`reject` 在 control 里不存在
3. **唯一的非局部跳出**：`exit`——立即终止当前 control 及其所有调用者
4. **所有错误处理必须显式写**

## 7.3 control 声明

```p4
control MyIngress(inout headers hdr,
                  inout metadata meta,
                  inout standard_metadata_t std_meta) {

    // 局部常量、变量、实例、动作、表
    bit<9> cpu_port = 255;

    action drop() { mark_to_drop(std_meta); }

    table port_fwd { /* ... */ }

    // 必须有一个 apply 块
    apply {
        if (hdr.ipv4.isValid()) {
            port_fwd.apply();
        }
    }
}
```

每个 control **必须有且仅有一个 `apply { ... }`**，这是它被调用时执行的入口。

## 7.4 控制块里能放什么

`apply` 之前可以声明：

| 元素 | 作用 |
| ---- | ---- |
| `const` 常量 | 编译期常量 |
| 局部变量 | 块作用域 |
| extern 对象实例 | `Counter(1024, ...) ctr;` |
| `action` | 局部动作 |
| `table` | 匹配-动作表 |
| 子控制块实例 | `InnerControl() inner;` |

> [!TIP]
> 如果一个 action / table 只给某个 control 用，**就声明在里面**——命名空间更干净，也方便读者理解依赖关系。

## 7.5 动作（action）

### 7.5.1 什么是动作

一个 action 是 **数据平面可直接执行的一段代码**。它能：

- 读写 headers、metadata
- 读控制平面写入的"动作数据"（参数）
- 调用 extern 方法（counter、register 等）

它 **不能**：

- 调用 table.apply
- 调用另一个 control 块
- 递归

### 7.5.2 声明语法

```p4
action Forward_a(out bit<9> outputPort, bit<9> port) {
    outputPort = port;
}
```

两种参数区分：

- **有方向参数**（`in`/`out`/`inout`）：由"当前 context"（通常是表或调用方）绑定
- **无方向参数**：代表 **动作数据**，由 **控制平面** 下发表项时提供，必须出现在参数列表末尾

### 7.5.3 调用方式

**① 隐式调用（表触发）**

表匹配命中时，表自动带上"控制平面填好的无方向参数"执行对应 action。

**② 显式调用**

从 control 的 `apply`、或从另一个 action 里直接调：

```p4
apply {
    Forward_a(std_meta.egress_spec, 9w1);
}
```

显式调用时 **所有参数** 都要提供——包括无方向参数。

## 7.6 `standard_metadata`（V1Model 快速入门）

在 V1Model 里，每个报文都配有一段 `standard_metadata_t`（来自 `v1model.p4`），是你和架构沟通的"仪表盘"。常用字段：

| 字段 | 方向 | 含义 |
| ---- | ---- | ---- |
| `ingress_port`   | 读 | 报文进入的物理端口 |
| `egress_spec`    | 写 | Ingress 要把报文送往的端口（关键字段） |
| `egress_port`    | 读 | Egress 阶段的端口（只读） |
| `packet_length`  | 读 | 入包字节数 |
| `instance_type`  | 读 | 包类型（正常 / 克隆 / 重新注入…） |
| `parser_error`   | 读 | parser 里 `verify` 失败时的错误码 |
| `deq_timedelta`  | 读 | 报文在队列中逗留时间（纳秒） |

写入 `egress_spec` 是最常见的操作——告诉架构："把这个包从这个端口送出去"。

> [!TIP]
> BMv2 约定 `egress_spec = 511` = drop。实际代码里用 `mark_to_drop(std_meta)`（v1model 预定义的 action）更清晰。

## 7.7 内置动作：`NoAction` 与 `mark_to_drop`

`core.p4` 定义的 `NoAction`：

```p4
action NoAction() { }
```

默认出现在所有表的动作列表中——表属性 `default_action` 不显式写的时候会用它。

`v1model.p4` 定义的 `mark_to_drop`：

```p4
@name("NoAction") action NoAction_0() { }
action mark_to_drop(inout standard_metadata_t smeta) {
    smeta.egress_spec = 511;   // 约定的 drop 端口
    smeta.mcast_grp  = 0;
}
```

常见用法：

```p4
action drop() { mark_to_drop(std_meta); }
```

## 7.8 子控制块：模块化你的代码

当 control 变复杂时，可以把一部分抽成子控制块：

```p4
control Acl(inout headers hdr, inout metadata m, inout standard_metadata_t s) {
    /* ... */
    apply { /* ACL 逻辑 */ }
}

control MainIngress(inout headers hdr, inout metadata m, inout standard_metadata_t s) {
    Acl() acl;

    apply {
        acl.apply(hdr, m, s);
        if (s.egress_spec == 511) return;  // ACL drop
        // 其他转发逻辑
    }
}
```

### 7.8.1 直接调用（语法糖）

对于 **没有构造参数** 的 control，可以不实例化直接调：

```p4
control Callee(/* 参数 */) { /* ... */ }

control Caller(/* 参数 */) {
    apply {
        Callee.apply(/* 参数 */);   // 等价于：Callee() c; c.apply(...);
    }
}
```

编译器会自动帮你生成一个局部实例。

### 7.8.2 带构造参数的 control

```p4
control Monitor(bit<32> threshold)(inout headers hdr, ...) { /* ... */ }

control Top(inout headers hdr, ...) {
    Monitor(1000) m1;   // 在实例化时传入构造参数
    apply {
        m1.apply(hdr, ...);
    }
}
```

**构造参数必须是无方向的**——因为它们在编译时（装配流水线时）固化。

## 7.9 在 control 中使用 `if` / `switch`

### 7.9.1 基于表的命中结果做分支

```p4
if (host_table.apply().hit) {
    // 命中
} else {
    lpm_table.apply();   // 未命中时退回 LPM
}
```

或：

```p4
if (host_table.apply().miss) {
    lpm_table.apply();
}
```

### 7.9.2 基于实际 action 做分支

```p4
switch (ingress_acl.apply().action_run) {
    deny:   { return; }
    permit: { /* ... */ }
    default: { /* ... */ }
}
```

这比"查表后根据某个字段判断"更安全——编译器能保证 label 都是表的合法 action。

## 7.10 `exit` 与 `return`

- `return` 终止当前 control（仅当前的）
- `exit` 终止当前 control **以及所有调用它的 control**

典型"短路"写法：

```p4
apply {
    ipv4_match.apply();
    if (std_meta.egress_spec == 511) return;   // 已 drop，后面不做了
    check_ttl.apply();
    if (std_meta.egress_spec == CPU_PORT) return;
    // ...
}
```

## 7.11 pipeline 全貌（V1Model）

参考一下 V1Model 下 control 块是如何"串"起来的——这决定了你要把逻辑写在哪里：

```text
Packet
  │
  ▼  (parser)
Parser ─────────────────┐
  │                      │
  ▼                      │ parser_error
VerifyChecksum           │
  │                      │
  ▼                      ▼
Ingress ←────── headers + metadata + std_meta
  │ egress_spec
  ▼
Traffic Manager (架构内建，不可编程：丢弃/复制/排队)
  │
  ▼
Egress
  │
  ▼
ComputeChecksum
  │
  ▼
Deparser ──► 出包
```

- **Ingress**：最主要的决策点——查表、改包头、设定 egress_spec
- **Egress**：常用于出方向修改（比如 VLAN push、镜像后处理）
- **Verify / Compute Checksum**：专门做校验和的块，可为空

## 7.12 完整示例：Ingress 骨架

```p4
control MyIngress(inout headers hdr,
                  inout metadata meta,
                  inout standard_metadata_t std_meta) {

    action drop()                { mark_to_drop(std_meta); }

    action set_egress(bit<9> port) {
        std_meta.egress_spec = port;
    }

    table dmac_forward {
        key     = { hdr.ethernet.dst: exact; }
        actions = { set_egress; drop; NoAction; }
        size    = 1024;
        default_action = NoAction;
    }

    apply {
        if (std_meta.parser_error != error.NoError) {
            drop();
            return;
        }
        if (hdr.ethernet.isValid()) {
            dmac_forward.apply();
        }
    }
}
```

## 7.13 本章小结

- `control` 块 = 数据平面的决策单元，必须有一个 `apply`
- `action` 有"有方向参数"（上下文绑定）和"无方向参数"（控制平面下发的动作数据）
- `mark_to_drop` 和 `NoAction` 是 V1Model 最常用的内置动作
- 可以嵌套调用子 control，用 `Callee.apply(...)` 直接调用无构造参数版本
- `return` 跳出当前 control，`exit` 跳出所有 control
- Ingress 是主战场，Egress 做出方向处理
